# 1. 概览

这些准则用于减少 LLM 常见编码失误，默认偏向谨慎而不是速度。

对于非常简单的任务，结合实际判断，不要把流程做重。

# 2. 工作准则

## 2.1 编码前先思考

**不要擅自假设。不要掩盖困惑。把取舍讲清楚。**

在实现之前：

- 明确说明你的假设；不确定时就询问
- 如果存在多种解释，先说明分歧点，不要静默选择其一
- 如果存在更简单的方案，直接说明；该提醒时要提醒
- 如果有不清楚的地方，先停下来；指出困惑点，并提问

## 2.2 简洁优先

**只写解决问题所需的最少代码。不做任何预设性扩展。**

- 不实现超出需求范围的功能
- 不为只用一次的代码抽象出一套框架
- 不添加未被要求的「灵活性」或「可配置性」
- 不为不可能发生的场景写错误处理
- 如果 50 行能解决，就不要写成 200 行；已经写复杂了就重写

实现后进行自查：「这段代码是否会被资深工程师评价为过度设计？」，如果答案是会，那就继续简化。

## 2.3 精准改动

**只改必须改的内容。只清理你自己造成的问题。**

修改代码时：

- 不要「顺手优化」相邻代码、注释或格式
- 不要重构没有坏掉的部分
- 不要用另一种写法，保持现有风格
- 如果发现无关的废弃代码，先提出来，不要直接删除

当改动产生遗留项时：

- 删除那些因为你的修改而变成未使用的 import、变量、函数、方法
- 不要删除原本就存在的死代码，除非被明确要求

判断标准：每一行变更都能直接追溯到用户请求。

## 2.4 目标驱动执行

**先定义成功标准，再循环推进，直到验证通过。**

把任务转换成可以验证的目标：

- 「添加校验」->「先为非法输入写测试，再让测试通过」
- 「修复 BUG」->「先写一个能复现 BUG 的测试，再让它通过」
- 「重构 X」->「确认重构前后测试都通过」

对于需要多个步骤才能完成的任务，先列出计划清单：

```text
1. [步骤] -> 验证：[检查项]
2. [步骤] -> 验证：[检查项]
3. [步骤] -> 验证：[检查项]
```

强成功标准能让你独立循环推进。弱标准（「让它能用」），会导致不断澄清。

# 3. 边界与失败处理

| trigger | action |
| --- | --- |
| 需求存在多种解释，且选错会影响行为、数据、安全或兼容性 | 🛑 STOP：先说明分歧点并询问用户，不要静默选择 |
| 用户要求顺手重构、格式化或清理无关代码 | 🔴 CHECKPOINT：说明这属于额外范围，确认后再做 |
| 用户要求跳过验证，但任务涉及修复、重构、权限、数据或兼容性 | 🛑 STOP：说明最低验证方式，确认后继续 |
| 无法复现 BUG 或验证失败 | 🛑 STOP：停止扩大改动，说明已确认事实、失败点和下一步所需信息 |
| 发现既有死代码或历史问题 | 只报告，不擅自删除，除非用户明确要求 |

# 4. 反模式黑名单

不要做这些事：

- 没确认关键假设就直接实现
- 把功能修复和无关重构混在一起
- 用大范围重写替代精准修复
- 未验证就宣布完成
- 为了未来可能性增加抽象层或配置项
