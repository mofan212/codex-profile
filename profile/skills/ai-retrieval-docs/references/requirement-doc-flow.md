# 1. 需求文档驱动流程

当触发时指定了需求文档，按本文件处理。

# 2. 项目范围

如果用户显式指定了项目或多个项目，以用户指定项目为准；否则以需求文档所在项目为主。

需要跨项目确认调用链时，可以读取同一 workspace 下兄弟项目。跨项目修改必须先说明范围并获得确认。

# 3. 收集上下文

按顺序收集：

- 需求文档所在目录文件列表。
- 需求文档内容。
- 同目录已有 AI 上下文入口、AI 检索文档、需求设计文档、示例数据的文件名；只读取与本需求直接相关的文档。
- 目标项目的 `git status` 和 `git diff`。
- diff 涉及的源码、测试、配置和文档。
- 通过 `rg` 检索到的入口类、处理器、工具类、错误码、常量、枚举、配置项、模板和测试。

如果当前 git 变更与需求文档不在同一项目，先判断是否属于同一 workspace 的兄弟项目；只读取跨项目上下文，修改跨项目文档前必须获得用户确认。

# 4. 生成或更新 AI 检索文档

把 AI 检索文档保存到需求文档同级目录下。默认命名：

- 如果需求文档形如 `10-xxx-需求文档.md`，优先使用相邻编号，例如 `11-xxx-AI检索说明.md`。
- 如果同主题已有 AI 检索文档，更新已有文档，不重复创建。
- 如果没有编号，使用 `<需求主题>-AI检索说明.md`。

同一主题的需求文档、设计文档和 AI 检索说明应能从文件名看出关系。重命名文档后，同步更新同目录内相对链接。

写入前按 [SKILL.md](../SKILL.md) 的「按需读取参考」读取对应写作参考。

# 5. 维护需求目录入口

如果需求文档所在目录存在多个需求文档、多个 AI 检索文档，或同时存在需求文档、设计文档、示例数据、AI 检索说明等容易混读的材料，在该目录维护 `00-AI上下文入口.md`：

- 已存在则更新。
- 不存在则创建。
- 内容只做加载路由和关键事实提示，不替代 AI 检索文档。

简单目录只有一个需求文档和一个 AI 检索文档时，可以不创建需求目录入口，除非用户要求。

# 6. 维护上层入口

从需求文档所在目录向上搜索：

1. 先判断当前目录是否为需求目录。需求目录通常包含需求文档、AI 检索文档、示例数据、设计文档或同主题材料。
2. 如果是需求目录，继续向上搜索，直到项目根、文档根或能力域根。
3. 查找沿途是否存在顶层 `00-AI上下文入口.md`、`AI上下文入口.md` 或 `AI检索入口.md`。
4. 如果存在，维护该入口，让它能路由到当前需求目录或能力域。
5. 如果不存在，但当前文档树明显需要渐进式加载入口，创建最合适层级的 `00-AI上下文入口.md`。优先放在需求目录的上层文档根或能力域根，而不是每个需求目录都放顶层入口。

维护入口前按 [SKILL.md](../SKILL.md) 的「按需读取参考」读取对应写作参考。

# 7. 不更新的情况

出现以下情况时，不要强行写文档：

- 需求文档只是草稿，且没有足够可验证事实来源支撑已实现事实。
- 同目录已有 AI 检索文档，但当前变更没有改变稳定代码事实、验证方式或加载路由。
- 用户明确只要求分析，不要求维护文档。

不更新时说明已读取依据和缺少的事实。
