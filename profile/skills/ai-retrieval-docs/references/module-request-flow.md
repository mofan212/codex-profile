# 1. 模块请求驱动流程

当触发时未指定文档，但指定了模块、功能、能力域、接口、类名、包名或业务流程，并要求生成、创建、维护、更新、补充或检查是否过期的 AI 检索文档、AI 检索说明或 AI 排查文档，按本文件处理。

纯粹查找 AI 检索文档、AI 检索说明或 AI 排查文档不是维护任务，不按本文件处理。

用户提到模块、功能、能力域、接口、类名、包名或业务流程，但没有明确要求维护、生成、创建、更新、补充或检查是否过期 AI 检索类文档时，不按本文件处理；按普通查找或分析处理，必要时询问是否转为 AI 检索文档维护任务。

如果执行中发现当前输入实际属于指定文档输入，例如文档标题经检索后可唯一匹配，停止本流程；该输入不属于模块请求驱动流程，不在本文件内判定文档类型。

# 2. 项目和目标范围

如果用户显式指定了项目或多个项目，以用户指定项目为准；否则以当前项目为主。

定义 `target_name` 为用户指定的模块、功能、能力域、接口、类名、包名或业务流程；不要把测试框架、评估模式、agent 角色、只读约束或 runner 标签当作 `target_name`、项目目录、文档根或目标文件候选。如果 `target_name` 对应多个模块或能力域，列出候选和依据，询问用户缩小范围。

将用户要求归一为 `requested_action`：

| signal | requested_action |
| --- | --- |
| 要求检查、确认、判断是否过期，只要求报告结论 | `check_only` |
| 要求更新、维护、补充、修正、同步已有文档 | `update` |
| 要求生成、创建、新增文档 | `create` |
| 同时要求检查并修复、检查后更新，或动作不明确 | 先询问用户确认动作 |

需要跨项目确认调用链时，可以读取同一 workspace 下兄弟项目。跨项目修改必须先说明范围并获得确认。

# 3. 收集上下文

自动继续条件仅限 `requested_action` 为 `check_only`，且只找到一个强匹配 AI 检索文档。未满足自动继续条件前，只收集足以定位候选文档和建议下一步的上下文，不要完整读取到可写文档的程度。

按顺序收集：

- 项目根、文档根和常见文档目录的文件列表
- AI 上下文入口、AI 检索入口中与 `target_name`、能力域或候选目录相关的路由
- 现有 AI 检索文档的文件名、标题和必要少量内容
- 通过 `target_name` 及其常见同义词检索到的目录名、包名、类名、接口路径、配置项、测试名和文档标题

模块代码与文档根定位失败按下表处理：

| failure | first_action | fallback | stop_when |
| --- | --- | --- | --- |
| 无法定位模块代码 | 用 `target_name` 及其同义词检索类名、包名、接口路径、测试名和配置项 | 列出已检索关键词和目录，询问用户提供模块代码路径 | 用户未提供路径，且没有任何可验证代码事实 |
| 无法定位文档根 | 查看项目根、README、`docs`、`doc`、`文档`、`需求` 目录和入口文件候选 | 给出候选目录和缺失依据，询问用户确认文档根 | 用户未确认文档根，且不存在可判断的 AI 检索文档目录 |

# 4. 查找候选 AI 检索文档

创建、更新或检查前必须查重。匹配范围包括文件名、标题、入口文档路由、文档内容、代码路径、接口、配置和测试。

强匹配：

| signal | requirement |
| --- | --- |
| 文件名或标题直接包含 `target_name` | 能明确指向目标模块或功能 |
| 入口路由明确把目标模块或功能指向某文档 | 该文档为强匹配候选 |
| 文档主要内容覆盖已定位模块代码路径、接口、配置或测试 | 不是零散提及，而是主体内容对应 |

弱匹配：

| signal | requirement |
| --- | --- |
| 文档内容零散提到目标模块关键词 | 只能说明相关，不能证明覆盖目标模块 |
| 入口路由只指向宽泛目录或能力域 | 只能作为候选范围 |
| 文档覆盖上游或下游相关流程 | 不能证明覆盖目标模块本身 |

按候选结果处理：

| candidate_result | action |
| --- | --- |
| 只有一个强匹配，且 `requested_action` 为 `check_only` | 告知文档路径和匹配依据，继续只读检查；不写文件 |
| 只有一个强匹配，且 `requested_action` 为 `update` | 🔴 CHECKPOINT：询问用户是否更新该文档，并说明是否可能同步同级或上层入口 |
| 只有一个强匹配，且 `requested_action` 为 `create` | 🔴 CHECKPOINT：询问用户是改为更新现有文档，还是确认另建新文档 |
| 存在多个强匹配 | 🔴 CHECKPOINT：列出候选、匹配依据和预计写入范围，等待用户选择目标文档和动作 |
| 只有弱匹配候选 | 🔴 CHECKPOINT：列出弱匹配依据，等待用户选择更新、检查、创建或指定其他文档 |
| 没有候选文档 | 🔴 CHECKPOINT：说明已检查范围，给出建议创建路径或需要用户补充的信息；确认前不创建 |

除「唯一强匹配 + 只读检查」外，不自动选择、不自动维护、不自动检查、不另建同主题文档。

# 5. 询问下一步

询问时说明已识别的 `target_name`、读取范围、候选文档、匹配依据和建议路径。涉及写入时，必须列出预计写入范围：目标 AI 检索文档，以及是否需要同步当前项目或当前文档树内的同级入口或上层入口。

下一步动作使用稳定问法：

| option | action |
| --- | --- |
| 更新现有文档 | 用户确认目标文档和入口写入范围后再维护 |
| 检查是否过期 | 用户选择目标文档后只检查并报告结论，不写文件；唯一强匹配时可直接继续 |
| 创建新文档 | 先建议目标路径并要求用户确认；确认前不写入 |
| 指定其他文档 | 用户提供文件路径、明确文件名或唯一标题后，停止本流程；按指定文档输入处理 |
| 取消 | 不写文件 |

如果没有候选文档，说明已检查的范围，并给出建议创建路径或需要用户补充的信息。如果 `requested_action` 为 `check_only` 且没有任何候选文档，告知没有可检查的 AI 检索文档，询问是否改为创建新文档并确认目标路径。

# 6. 用户确认后

用户确认目标文件和动作后再继续。唯一强匹配且 `requested_action` 为 `check_only` 时，可视为目标文件和动作已由请求与候选结果共同确认。

| confirmed_action | requirement |
| --- | --- |
| 更新现有文档 | 读取目标文档、模块源码、测试、配置、必要 diff 和入口；如果文档大面积过期、模块经历重构或新增核心调用链，按 [writing-ai-retrieval-doc.md](writing-ai-retrieval-doc.md) 重新理解业务语义并筛选可写入事实；只更新确认的目标文档和已告知的必要入口 |
| 检查是否过期 | 对照代码和当前变更报告过期点、代码依据和建议更新位置；不写文件 |
| 创建新文档 | 确认目标路径后读取写作所需事实来源，按 [writing-ai-retrieval-doc.md](writing-ai-retrieval-doc.md) 理解业务语义、筛选可写入事实并组织文档结构，再创建目标文档 |

当前项目或当前文档树内必要入口只能在已告知写入范围后做最小更新；跨项目或当前工作区外写入仍必须先获得确认。

文档内容必须聚焦已确认模块或功能：

- 写清代码位置、入口类或接口、执行链路、关键配置、兼容边界、验证命令和排查关键词
- 区分已实现事实、当前变更事实、用户明确结论和待确认信息
- 不要把需求设想、模块名称推测或临时讨论写成已实现事实

# 7. 不更新的情况

出现以下情况时，不要强行写文档：

- 用户未确认目标文件或下一步动作
- 无法定位模块代码或文档根
- 缺少可验证事实来源
- 用户只是纯查找或明确只要求分析

不更新时说明已读取依据、候选文档或建议路径，以及没有写入的原因。
