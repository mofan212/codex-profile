# 1. AI 检索文档写作要求

本文件只在准备生成或更新 AI 检索文档时读取。

本文件负责约束：

| responsibility | requirement |
| --- | --- |
| 写入前理解 | 确认模块业务语义、调用关系、核心对象和边界条件 |
| 事实筛选 | 只写入长期有效且有代码、测试、配置、diff、已有文档或用户明确结论支撑的事实 |
| 文档结构 | 优先写清业务语义、调用地图、执行链路、兼容边界和排查关键词 |
| 不写内容 | 不写临时过程、需求设想、未验证推测和已废弃方案展开细节 |

# 2. 写入前理解

创建新 AI 检索文档，或在文档大面积过期、模块经历重构、新增核心调用链时更新既有文档，先完成业务语义理解。

理解提示：

```markdown
上移一层抽象，使用项目领域词汇画出相关模块、调用方、被调方、入口、核心数据对象和业务规则，再回到代码证据确认哪些可以写入长期事实。
```

按以下顺序确认：

- 先从入口、调用方、被调方和配置找出模块在系统中的位置
- 用项目已有领域术语描述业务角色、业务动作、状态变化和边界条件
- 区分本模块职责、上游输入、下游影响、复用点和非职责范围
- 对照源码、测试、配置和已有文档确认每个业务结论
- 把无法从事实来源确认但影响写作的信息标为待确认

写入时优先落到：

- 文档定位说明它解决哪个模块或功能的后续检索问题
- 核心结论先给业务语义和调用地图，再展开代码位置
- 执行入口链路覆盖用户入口、内部处理、下游调用和关键分支
- 兼容边界记录默认路径、例外路径、开关、兜底和复用限制

# 3. 推荐结构

根据需求实际裁剪，优先使用：

- `# 1. 文档定位`
- `# 2. 核心结论`
- `# 3. 主要代码位置`
- `# 4. 执行入口链路`
- 领域行为章节，例如状态流转规则、转换规则、上下文字段、错误处理、持久化行为、元数据规则
- 兼容性和边界章节
- `# N. 当前测试重点` 或 `# N. 验证建议`
- `# N. AI 快速检索提示`

需要更完整模板时，读取 [document-template.md](document-template.md)。

写入或更新完成后，读取 [markdown-and-checklist.md](markdown-and-checklist.md) 进行检查。

# 4. 写作原则

- 面向后续排查和理解，不写成发布说明、汇报材料或产品介绍
- 多写类名、方法名、常量名、文件路径、行为边界和验证命令
- 记录隐藏规则和兼容性要求，例如默认值、开关、兜底逻辑、跳过分支、复用边界
- 避免复制大段需求文档或代码
- 不要编造未从需求、已有文档、代码、测试、diff 或用户明确结论中确认的行为
- 不确定但重要的信息要明确标注，不要把推测写成事实
- 优先写稳定事实，不写临时调试过程、未合并方案或个人判断
- 旧逻辑、新逻辑和兼容行为同时存在时，明确说明默认路径和触发条件

# 5. 沉淀边界

需求文档和 AI 检索文档必须分层维护：

- 需求文档记录本次任务过程、需求澄清、PRD、issue 拆分、review 过程、临时注意事项和被否决方案
- AI 检索文档记录完成后仍然成立的长期事实，例如业务规则、代码入口、执行链路、状态流转、关键配置、兼容边界、验证命令和排查关键词

写入 AI 检索文档前，对每条信息做以下判断，如果不满足判断，则不写入文档：

- 三个月后未来 AI 处理相关问题时，是否仍然需要这条信息。

不要写入以下内容：

- 需求讨论过程、澄清对话原文和 PRD 正文
- review 中间意见、临时 TODO、调试记录和执行流水
- 已废弃方案的展开细节，除非它解释了仍然存在的兼容边界或当前实现约束
- 没有代码、测试、diff、已有文档或用户明确结论支撑的个人判断

如果某条信息只解释本次任务如何推进，保留在需求文档；如果它描述系统完成后的真实行为、入口或边界，才写入 AI 检索文档。

# 6. 风格参考

优先参考所在项目内已有的高质量 AI 检索文档，但不要把某个项目的目录名、模块名或历史需求写进通用规则。

评价参考文档是否可借鉴时，优先看这些特征：

- 是否清楚描述稳定代码事实、执行链路、兼容边界、验证方式和检索关键词
- 是否区分已实现事实、需求目标和不确定信息
- 是否避免复制需求正文、代码大段内容和临时讨论过程
