# 1. AI 检索文档写作要求

本文件只在准备生成或更新 AI 检索文档时读取。

本文件负责约束：

| responsibility | requirement |
| --- | --- |
| 写入前理解 | 确认模块业务语义、检索入口、关键对象和容易误判的边界 |
| 事实筛选 | 只写入长期有效、有事实来源支撑，且能减少后续搜索、定位或重复推理成本的信息 |
| 文档结构 | 优先组织成索引、关键结论、隐藏规则、兼容边界、验证入口和排查关键词 |
| 不写内容 | 不写代码说明书、完整调用链解释、临时过程、需求设想、未验证推测和已废弃方案展开细节 |

# 2. 写入前理解

创建新 AI 检索文档，或在文档大面积过期、模块经历重构、新增核心链路时更新既有文档，先完成业务语义理解。

业务抽象提示词：

```markdown
上移一层抽象，使用项目领域词汇找出相关模块、调用方、被调方、入口、核心数据对象和业务规则，再回到代码证据确认哪些信息值得作为检索索引长期保留。
```

按以下顺序确认：

- 先从入口、调用方、被调方和配置找出模块在系统中的位置
- 用项目已有领域术语描述业务角色、业务动作、状态变化和边界条件
- 区分本模块职责、上游输入、下游影响、复用点和非职责范围
- 对照源码、测试、配置和已有文档确认每个业务结论
- 把无法从事实来源确认但影响写作的信息标为待确认

写入时优先落到：

- 文档定位说明它解决哪个模块或功能的后续检索问题
- 检索索引给出入口类、关键方法、配置项、常量、测试或跨项目位置
- 关键结论记录源码不容易一眼看出的业务语义、隐藏规则和误判边界
- 关键链路只保留跨模块跳转、隐藏分支、默认路径、例外路径、开关、兜底和复用限制

# 3. 写入门槛

AI 检索文档不是实现说明书。每条新增信息 **必须同时满足**：

- 三个月后仍可能影响相关问题的搜索、定位、判断或排查
- 有代码、测试、配置、diff、已有文档或用户明确结论支撑
- 比直接读取源码更节省上下文，或能避免后续 AI 反复推理同一隐含结论
- 每条只承担一个检索价值，不在多个章节重复展开同一事实

新增信息 **还至少命中以下一类** 价值：

| value | write_when |
| --- | --- |
| `search_index` | 给出后续检索需要的入口类、方法、常量、配置项、错误码、测试名、文件路径或跨项目位置 |
| `hidden_rule` | 记录源码分散、命名不直观、需要多步推理才能确认的隐藏规则 |
| `compat_boundary` | 记录默认路径、旧逻辑、新逻辑、开关、兜底、跳过分支或复用限制 |
| `diagnosis_entry` | 给出排查入口、关键验证命令或最小复现检查点 |
| `misread_guard` | 避免把相邻能力、旧文档、生成逻辑、运行时逻辑或同名概念混读 |

如果信息只是解释某个类或方法「做了什么」，且后续 AI 能通过索引快速读到源码，不写正文，只保留必要关键词或路径。

# 4. 推荐结构

根据需求实际裁剪，优先使用：

- `# 1. 文档定位`
- `# 2. 快速检索索引`
- `# 3. 关键结论`
- `# 4. 关键链路`
- `# N. 兼容性和边界`
- `# N. 验证入口`

`关键链路` 只在链路跨模块、跨项目、有隐藏分支或容易误判时保留；普通顺序调用不展开逐步职责，改写为检索索引。

需要模板时，读取 [document-template.md](document-template.md)。

写入或更新完成后，读取 [completion-checklist.md](completion-checklist.md) 进行检查。

# 5. 写作原则

- 面向后续排查和理解，不写成发布说明、汇报材料或产品介绍
- 多写能定位源码的类名、方法名、常量名、文件路径、行为边界和验证命令，少写逐方法解释
- 记录隐藏规则和兼容性要求，例如默认值、开关、兜底逻辑、跳过分支、复用边界
- 复杂链路优先沉淀入口、跳转点、分支条件和误判点，细节回源码确认
- 测试只保留代表性测试、验证命令和覆盖意图，不列完整测试清单
- 排查内容只保留最短路径、关键断点和判断顺序，不写成操作手册
- 避免复制大段需求文档或代码
- 不要编造未从需求、已有文档、代码、测试、diff 或用户明确结论中确认的行为
- 不确定但重要的信息要明确标注，不要把推测写成事实
- 优先写稳定事实，不写临时调试过程、未合并方案或个人判断
- 旧逻辑、新逻辑和兼容行为同时存在时，明确说明默认路径和触发条件

# 6. 更新策略

更新既有 AI 检索文档时，默认先压缩再追加：

| trigger | action | forbidden |
| --- | --- | --- |
| 稳定事实变化 | 替换旧事实，并删除与新事实冲突的旧描述 | 在旧描述后追加新段落导致新旧并存 |
| 同类入口、关键词或边界重复 | 合并为一个索引项或一组短列表 | 重复解释同一链路 |
| 代码可直接定位且规则不隐藏 | 保留路径、类名、方法名或关键词 | 把源码逻辑改写成长段说明 |
| 测试或验证方式增加 | 保留最小验证命令和代表性测试 | 罗列全部测试方法 |
| 文档超过检索需要 | 删除过程性解释、逐方法职责和重复背景 | 继续追加章节 |

创建新文档时，默认以短索引文档为目标；复杂跨项目链路可以放宽篇幅，但每个章节都应能说明它为什么不能由入口索引和源码快速替代。

# 7. 沉淀边界

需求文档和 AI 检索文档必须分层维护：

- 需求文档记录本次任务过程、需求澄清、PRD、issue 拆分、review 过程、临时注意事项和被否决方案
- AI 检索文档记录完成后仍然成立，且能帮助后续 AI 更快检索、判断或排查的长期事实，例如代码入口、隐藏规则、关键配置、兼容边界、验证入口和排查关键词

不要写入以下内容：

- 需求讨论过程、澄清对话原文和 PRD 正文
- review 中间意见、临时 TODO、调试记录和执行流水
- 已废弃方案的展开细节，除非它解释了仍然存在的兼容边界或当前实现约束
- 完整调用链逐步说明，除非链路本身跨模块、跨项目或存在隐藏分支
- 普通类职责、普通方法职责和可由类名方法名直接理解的信息
- 完整测试方法清单、完整排查手册和实现沉淀原文
- 没有代码、测试、diff、已有文档或用户明确结论支撑的个人判断

如果某条信息只解释本次任务如何推进，保留在需求文档；如果它描述系统完成后的真实行为、入口或边界，才写入 AI 检索文档。

# 8. 风格参考

优先参考所在项目内已有的高质量 AI 检索文档，但不要把某个项目的目录名、模块名或历史需求写进通用规则。

评价参考文档是否可借鉴时，优先看这些特征：

- 是否用较短篇幅给出稳定事实、关键入口、兼容边界、验证入口和检索关键词
- 是否把复杂实现压缩为索引、误判边界和隐藏规则，而不是逐步解释源码
- 是否区分已实现事实、需求目标和不确定信息
- 是否避免复制需求正文、代码大段内容和临时讨论过程
