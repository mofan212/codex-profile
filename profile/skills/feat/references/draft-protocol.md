# 1. 建档协议

显式调用 `feat` 进入 `draft_requirement` 时，按以下步骤执行：

1. 定位合适的文档目录
2. 用最多 15 个字概括用户描述的需求，作为需求目录名
3. 在文档目录下创建 `<需求目录名>/`
4. 计算需求序号
5. 在 `<需求目录名>/` 下创建 `<需求序号>-<需求目录名>-需求文档.md`
6. 读取 [requirement-template.md](requirement-template.md)，根据用户已提供信息填入草稿
7. 不臆造业务规则、验收标准、实现方案或代码事实；未知内容保留为空或写成待完善项

这里的「需求目录」不是固定名为 `需求`，而是基于用户需求概括出的短名目录，例如 `订单超时关闭/`。

# 2. 文档目录定位

按下表从上到下判断：

| priority | condition | action |
| --- | --- | --- |
| 1 | 用户明确指定目录 | 使用用户指定目录 |
| 2 | 已有同类需求文档，且能判断需求目录承载位置 | 如果文档位于独立需求目录内，在该需求目录的父目录下创建新需求目录；如果文档直接位于需求集合目录内，在该集合目录下创建新需求目录 |
| 3 | 只有一个 `docs/`、`doc/` 或 `documentation/` | 在该目录下创建需求目录 |
| 4 | 多个候选目录且无法判断主文档入口 | 🔴 CHECKPOINT：只问「需求文档应该放在哪个目录？」 |

无法判断已有需求文档属于独立需求目录还是平铺的需求集合目录时，按多个候选目录处理。不要为了继续流程自行选择第一个候选目录。

# 3. 需求目录命名规则

| rule | requirement |
| --- | --- |
| 来源 | 基于用户原始需求描述概括，不引入用户未提到的业务对象 |
| 长度 | 最多 15 个汉字；英文、数字和符号按实际字符计入 |
| 风格 | 使用简短名词短语，例如 `订单超时关闭`、`库存同步商城` |
| 禁止 | 不使用个人姓名、账号、机器路径、日期时间、阶段词或泛称，例如 `新需求`、`需求文档`、`功能开发` |
| 文件名 | 需求文档名必须是 `<需求序号>-<需求目录名>-需求文档.md` |

# 4. 需求序号规则

需求序号按当前文档目录及其一级子目录、同类需求目录中已有 `<编号>-*-需求文档.md` 递增；如果已有编号带前导零，则保持相同位数。没有已有编号时默认从 `01` 开始。无法可靠判断编号时，先问用户，不要自行猜测。
