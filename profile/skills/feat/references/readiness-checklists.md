# 1. Feature DoR

在调用 `to-issues` 前检查：

- 目标是否明确；
- 非目标是否明确；
- 业务规则是否没有关键歧义；
- 验收标准是否可验证；
- 影响模块是否已初步识别；
- 验证方式是否已说明；
- 关键风险是否已记录；
- 是否需要 ADR 已判断；
- 是否不存在阻塞拆分的未澄清问题。

如果任一关键项不满足，先补需求文档，不要拆 Issue。

# 2. Issue DoR

每个 Issue 开始实现前检查：

- 来源需求文档明确；
- 本 Issue 目标单一；
- 范围和非范围明确；
- 依赖关系明确；
- 验收标准明确；
- 验证方式明确；
- 预计改动位置可以定位；
- 没有必须先回答的开放问题。

# 3. Issue DoD

每个 Issue 完成前检查：

- 代码实现满足验收标准；
- 已完成必要测试或明确验证；
- 没有引入无关重构；
- Review 发现的问题已处理或有明确解释；
- 相关文档已按需更新；
- 如果产生新的代码事实，已更新 `实现沉淀记录.md`；
- 如果没有产生需要沉淀的事实，完成说明中明确写 `无需更新实现沉淀记录`。

# 4. 垂直切片检查

Issue 应按用户可感知或系统可验证的完整行为拆分。

避免这种技术层拆分：

- 新增数据库字段；
- 新增查询接口；
- 新增 Service；
- 新增测试。

优先拆成这种垂直切片：

- 支持识别满足条件的数据；
- 支持执行目标状态变更并保持幂等；
- 支持调度或入口触发完整流程；
- 补齐正式 AI 检索文档和排查入口。
