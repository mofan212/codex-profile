# 1. Feature DoR

在调用 `to-issues` 前检查。任一必需项不满足时，先补需求文档，不要拆 Issue。

| 检查项 | 必需性 | 通过证据 | 失败处理 |
| --- | --- | --- | --- |
| 目标明确 | 必需 | 需求文档能用一句话说明要达成的用户或系统结果。 | 补充目标描述。 |
| 非目标明确 | 必需 | 需求文档列出本次不做的能力、场景或边界。 | 补充非目标，避免 Issue 范围外扩。 |
| 业务规则无关键歧义 | 必需 | 关键判断条件、状态变化、异常分支和约束已有明确结论。 | 使用 `grill-with-docs` 继续澄清。 |
| 验收标准可验证 | 必需 | 每条验收标准都能对应页面行为、接口结果、数据状态、日志、测试或命令输出。 | 改写验收标准。 |
| 影响模块已初步识别 | 必需 | 已列出可能涉及的模块、文档目录、入口或核心代码区域。 | 读取必要项目文档或源码后补充。 |
| 验证方式已说明 | 必需 | 已说明测试、构建、手工检查、接口调用或其他验证方式。 | 补充验证建议。 |
| 关键风险已记录 | 必需 | 兼容性、数据迁移、性能、安全、发布或回滚风险已判断。 | 补充风险或说明无明显风险。 |
| ADR 需求已判断 | 必需 | 已说明是否存在长期技术决策；需要 ADR 时已记录。 | 判断并按项目规则补 ADR。 |
| 无阻塞拆分的开放问题 | 必需 | 剩余问题不会影响 Issue 边界、依赖或验收。 | 先回答开放问题。 |

# 2. Issue DoR

每个 Issue 开始实现前检查。任一必需项不满足时，先补 Issue，不要交接实现。

| 检查项 | 必需性 | 通过证据 | 失败处理 |
| --- | --- | --- | --- |
| 来源需求文档明确 | 必需 | Issue 链接或明确引用对应需求文档。 | 补充来源。 |
| Issue 目标单一 | 必需 | Issue 只描述一个可独立交付的用户可感知或系统可验证行为。 | 拆分或收窄 Issue。 |
| 范围和非范围明确 | 必需 | Issue 写清本次做什么、不做什么。 | 补齐边界。 |
| 依赖关系明确 | 必需 | 前置 Issue、配置、数据、外部系统或无依赖已说明。 | 补充依赖。 |
| 验收标准明确 | 必需 | 验收项能直接判断通过或失败。 | 改写验收标准。 |
| 验证方式明确 | 必需 | 已说明应运行的测试、构建、接口调用或人工检查。 | 补充验证方式。 |
| 预计改动位置可定位 | 必需 | 能定位到模块、目录、入口、配置或文档区域。 | 使用 `read-project-docs` 或源码搜索补充。 |
| 代码事实已校验 | 条件必需 | 进入代码实现、修改代码事实，或验收依赖当前源码、测试、配置或调用入口时，已确认 Issue 假设与当前代码事实一致；不一致处已记录并完成确认。 | 先完成代码事实校验，不要直接实现。 |
| 无必须先回答的开放问题 | 必需 | 剩余问题不会阻塞实现开始。 | 先澄清再实现。 |

# 3. Issue DoD

每个 Issue 完成前检查。任一必需项不满足时，不要进入下一 Issue 或最终归档。

| 检查项 | 必需性 | 通过证据 | 失败处理 |
| --- | --- | --- | --- |
| 实现满足验收标准 | 必需 | 每条验收标准都有对应实现证据。 | 继续修复或调整范围。 |
| 已完成必要验证 | 必需 | 已运行相关测试、构建、接口调用或人工检查；无法运行时说明原因和替代核对范围。 | 补验证或说明阻塞。 |
| 无无关重构 | 必需 | diff 只包含与 Issue 目标相关的代码、测试、配置和文档。 | 回退或拆出无关改动。 |
| Review 问题已回应 | 必需 | Review 发现的问题已修复或有明确不修理由；用户跳过 Review 时已记录原因和风险。 | 处理 Review，或补充跳过原因和风险。 |
| Review 退出条件满足 | 必需 | 最近一轮 Review 没有阻塞级问题，且非阻塞建议已处理或有明确不处理理由；或用户明确跳过 Review 且已记录原因和风险。 | 继续处理 Review 结果，或补充跳过原因和风险。 |
| 相关文档已按需更新 | 必需 | 需求、Issue、实现记录或 AI 检索资料已按实际影响更新。 | 补文档或说明无需更新。 |
| Issue 实现事实已沉淀 | 条件必需 | 产生入口、调用链、配置项、数据结构、验证命令、排查关键词等事实时，已更新 `.feat-tmp/issues/*-实现沉淀.md`。 | 更新需求文档同级目录下的 `.feat-tmp/issues/<需求序号>-<Issue序号>-<Issue简述>-实现沉淀.md`。 |
| 无需沉淀时已说明 | 条件必需 | 没有产生需沉淀事实时，完成说明包含 `无需更新 Issue 实现沉淀`。 | 补充完成说明。 |

# 4. 垂直切片检查

Issue 应按用户可感知或系统可验证的完整行为拆分。

| 类型 | 判定信号 | 处理 |
| --- | --- | --- |
| 技术层拆分 | 只描述新增数据库字段、查询接口、Service、Mapper、测试或配置。 | 避免作为独立 Issue，合并进能交付行为的垂直切片。 |
| 垂直切片 | 描述识别满足条件的数据、执行目标状态变更并保持幂等、调度或入口触发完整流程、补齐正式 AI 检索文档和排查入口。 | 优先作为 Issue 边界。 |

# 5. Review 结果处理规则

本 Skill 不规定 Subagents 或外部 Agent 如何执行 Review；Review 循环协议见 [review-loop.md](review-loop.md)。本节只规定 Codex 收到 Review 结果后的分类和处理门禁。

如果 Review 未标注严重级别，Codex 必须先分类，再决定处理方式。

| 类型 | 判定标准 | 处理规则 | 进入下一步条件 |
| --- | --- | --- | --- |
| 阻塞级问题 | 影响验收、核心业务正确性、数据一致性、兼容性、安全、发布风险、回归风险或验证可信度。 | 必须修复；无法修复时说明原因、影响和后续处理方式。 | 问题已修复，或暂停并确认后续处理方式。 |
| 非阻塞建议 | 不影响验收和正确性，只改善局部可读性、命名、结构或补充性测试，且不扩大 Issue 范围。 | 可处理；不处理时说明理由。 | 建议已处理，或已有明确不处理理由。 |
| 不处理 | 与项目约定冲突、缺少事实依据、超出当前 Issue 范围，或会引入无关重构。 | 不修改代码，只记录理由。 | 不影响 Issue DoD。 |
| 无问题 | 最近一轮 Review 没有阻塞级问题。 | 允许进入 Issue DoD。 | Review 门禁通过。 |
| 跳过 Review | 用户明确选择跳过 Review。 | 记录跳过原因和风险；不要称为独立 Review 或无阻塞 Review。 | 允许继续执行 Issue DoD 的其他检查。 |

# 6. `.feat-tmp/` 临时工作区归档规则

`.feat-tmp/` 目录是 `feat` 工作流运行期间的临时工作区，`.feat-tmp/issues/` 保存 Issue 实现沉淀材料。全部 Issue 完成后，必须先使用 `ai-retrieval-docs` 生成或更新正式 AI 检索文档，再按当前运行环境的文件删除、安全审批和跨项目修改规则删除 `.feat-tmp/`；如果需要审批，先请求确认。

删除 `.feat-tmp/` 前必须确认：

- 全部 Issue 已完成。
- 正式 AI 检索文档已生成或更新。
- AI 检索文档已覆盖 `.feat-tmp/` 中需要长期保留的入口、调用链、配置项、数据结构、验证命令和排查关键词。
- 删除范围仅限需求文档同级目录下的 `.feat-tmp/`。
- 删除动作符合当前运行环境的审批、沙箱和跨项目修改规则。
