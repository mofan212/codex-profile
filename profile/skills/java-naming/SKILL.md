# 1. Java 命名入口规则

使用本 Skill 为 Java 后端代码选择包路径、类名、方法名、字段名、变量名、常量名和角色后缀。

命名前先阅读当前代码上下文，优先遵循所在仓库、模块、同级包和相邻类的既有风格。不要在职责没有确定前先套后缀。

可以在语义明确时引入 `Dto` 和 `Dao` 后缀；涉及这两个后缀时读取 [dto-dao-naming.md](references/dto-dao-naming.md)。不要默认引入 `VO`、`DO`、`PO`、`BO`、`POJO` 等其他对象后缀命名体系，除非用户特别说明，或仓库里已经存在这类名称且需要兼容。

# 3. 工作流程

| step | input | action | output |
| --- | --- | --- | --- |
| 1. 确定命名范围 | 用户问题、候选名称、涉及代码片段或文件路径 | 明确要命名或评审的是包、类、接口、方法、字段、变量还是常量 | `scope`：需要加载的引用文件和输出字段 |
| 2. 对齐本地风格 | `scope`、当前模块、同级包或相邻代码 | 识别业务域、职责边界和本地命名风格，优先查看同包、同模块、同职责对象 | `local_style`：说明是否存在稳定约定 |
| 3. 校验职责语义 | 候选名称、职责描述、`local_style` | 判断职责是否足够支撑后缀、动词或包层级；沿用稳定领域词拼写，例如 `OrderId` 和 `OrderID` 之间不要随意切换 | `naming_basis`：职责不足时列出缺失信息 |
| 4. 检查冲突复用 | 候选名称、同职责既有对象、相关引用文件 | 检查是否已有同职责包、类、接口、抽象基类、测试类、方法或常量 | `conflict_check`：说明复用、兼容或新增理由 |
| 5. 判断影响面与检查点 | 本地约定、本 Skill 通用规则、潜在公开契约影响 | 只评审或命名本次用户指定对象；判断是否需要暂停确认 | `checkpoint` 或最终命名结论 |
| 6. 输出命名结论 | 最终命名结论或评审结论 | 说明当前命名是否可接受、更推荐的命名、推荐理由和兼容理由 | 简短输出，字段只包含本次任务相关内容 |

# 4. 异常与分歧处理

| trigger | first_action | fallback | forbidden |
| --- | --- | --- | --- |
| 无法确认仓库包名前缀、组织名或模块层级 | 只给相对包路径或 `<root>` 占位路径，并说明待确认项 | 如果用户要求完整包路径，先暂停确认前缀、模块层级和目标包边界 | 臆造公司名、用户名、邮箱、机器路径或不存在的项目名 |
| 用户给出的职责不足以判断角色后缀 | 先列出缺失信息；基于已知事实给出条件式候选和适用条件 | 如果用户要求最终命名或直接落地，触发 CHECKPOINT，不继续定名 | 把条件式候选当作最终命名，或直接套 `Service`、`Manager`、`Helper`、`Util` 等泛后缀 |
| 本 Skill 规则与仓库既有稳定风格冲突 | 优先保留本地一致性，并在 `compatibility` 中说明兼容理由 | 如果用户要求改掉稳定风格，先列出影响面并暂停确认 | 为了通用规则推动批量改名 |
| 用户要求新增对象后缀体系 | 检查仓库是否已有稳定体系；没有时说明不推荐，并给出 `Dto`、领域对象、实体或 `Dao` 替代 | 如果用户坚持新增体系，先确认命名边界、迁移范围和兼容策略 | 默认扩展 `VO`、`DO`、`PO`、`BO`、`POJO` 等对象后缀体系 |
| 名称可能影响公开 API、数据库映射、序列化字段、配置键或外部协议 | 先指出影响面，再给出兼容保留或迁移命名建议 | 信息不足时输出「仅命名建议，不构成迁移方案」，并列出需要确认的契约点 | 只按代码风格给出重命名结论 |

# 5. 必须暂停确认的场景

满足以下任一条件时，触发 🔴 CHECKPOINT，先向用户确认，不继续给出最终命名或落地式重命名方案：

- 需要批量重命名既有包、类、方法、字段或常量
- 用户要求按本 Skill 通用规则改掉仓库既有稳定命名体系，且会影响公开 API、数据库映射、序列化字段或配置键
- 用户要求新增 `VO`、`DO`、`PO`、`BO`、`POJO` 等后缀体系，但当前仓库没有稳定先例
- 命名对象职责不足以判断后缀，且用户要求给出最终命名或直接落地，例如只说「写个工具类」「写个服务」「写个管理类」

# 6. 决策原则

## 6.1 优先级

命名优先级从高到低为：

1. 仓库或模块已有强约定
2. 同级包、相邻类和相邻成员的稳定风格
3. 本 Skill 的通用规则和对应引用文件

当本 Skill 与仓库既有风格冲突时，优先保留本地一致性，并在回答中说明这是兼容决策。

## 6.2 反模式黑名单

避免以下命名：

- 使用拼音，或把拼音和英文混用
- 用难以理解的缩写替代清晰单词
- 使用特殊字符，常量中的下划线除外
- 使用 Java 关键字、JDK 类名或主流框架核心类名作为自定义类型名
- 为了追求短名称牺牲业务含义
- 用 `for`、`to`、`from`、`with`、`of` 等介词时写成 `4`、`2` 这类谐音缩写
- 用 `Manager`、`Service`、`Helper`、`Util` 掩盖真实职责
- 未确认上下文就设计完整包路径
- 为了统一风格主动改既有命名
- 默认新增分层后缀体系

# 7. 输出格式

默认使用简短列表，只包含本次任务相关字段：`current`、`verdict`、`recommendation`、`reason`、`compatibility`、`scope`。没有当前命名时省略 `current`；`scope` 使用 `package`、`class`、`method`、`field`、`constant` 等命名对象类型。

```yaml
- current: UserManager
  verdict: 不推荐
  recommendation: UserPermissionService
  reason: 当前职责是用户权限判断和授权编排，`Manager` 无法表达具体职责
  compatibility: 如果已作为公开 Bean 或外部契约使用，改名时需要保留兼容入口
```
