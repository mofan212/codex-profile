# 2. Java 实现规则

## 2.1 结构与命名

- 类、方法、参数和字段名称要表达业务含义，职责保持单一；不要扩大接口行为面，或用宽泛名称掩盖职责
- 任务类、调度类、入口类、Controller 或 Service 编排方法优先只做流程编排；复杂对象构造、字段填充和分组转换下沉到专门构造类或工厂类
- 只有存在真实 `is-a` 语义时才使用继承；不要为了复用静态工具能力引入语义不成立的继承
- 除非存在同名类冲突，Java 正文引用类型统一在文件头部使用 `import`，不要在正文中书写类的全限定名

## 2.2 兼容与依赖

- 新代码不要依赖 `@Deprecated` 类、方法或职责偏移的历史工具类，优先改为依赖非废弃的稳定入口。旧依赖只是兼容入口且只做静态转发时，可以保留并说明依据
- 需要兼容旧调用点时，让旧类静态转发到新的稳定入口，不要让新代码反向依赖旧兼容类
- 必须复用废弃类中的常量、判断逻辑或命名规则时，先抽取到非废弃的稳定领域工具类，新代码依赖稳定入口，旧类只保留兼容转发
- Spring Bean、基础设施协作者或稳定运行时依赖不应作为普通业务参数在多层方法间传递；优先注入到当前组件或专门协作者中，方法参数只保留本次调用真正变化的业务数据
- 使用 Stream、Optional、lambda 等 Java 8 特性时，以可读性优先，不为展示技巧牺牲直接表达

## 2.3 集合操作风格

处理集合时，优先使用 Apache Commons Collections4 的工具类完成空值兜底、判空、过滤、转换等通用操作；项目已有等价封装时，优先遵循项目既有封装。

对简单集合逻辑，优先使用 Stream 表达查询、过滤、映射、匹配、聚合等意图，避免展开成冗长的 `for` 循环、临时变量和多层 `if`。

| trigger | requirement | forbidden |
| --- | --- | --- |
| 集合可能为 `null` 后继续遍历或 stream | 优先使用 `ListUtils.emptyIfNull(...)`、`CollectionUtils.emptyIfNull(...)` 等 Commons Collections4 工具类归一化为空集合 | 手写重复的空值和空集合判断，除非提前返回能显著提升可读性 |
| 简单筛选、映射、取首个、任意匹配、全部匹配 | 优先使用 Stream 的 `filter`、`map`、`findFirst`、`anyMatch`、`allMatch` 等表达主流程 | 为简单查询写命令式循环、`continue` 和临时变量 |
| Stream 中只是调用已有无参方法或静态方法 | 优先使用方法引用 | 写成无必要的 lambda |
| Stream 条件需要组合外部变量、工具类调用或业务判断 | 使用 lambda，并保持条件短而直接 | 为追求方法引用强行拆出低价值私有方法 |
| 逻辑包含复杂分支、副作用、日志、异常处理、多步状态变更 | 保留命令式写法，优先保证控制流清晰 | 为了 Stream 风格压缩复杂流程 |

## 2.4 抽象与参数收敛

- 参数收敛必须改善语义或调用边界；不要只为了减少参数数量引入值对象或配置对象

## 2.5 Web、MyBatis 与日志

- 新增或修改 Controller、接口参数、权限或业务注解时，如果项目已有扩展业务注解、权限注解或参数描述注解，沿用既有约定；不要在没有现有模式的项目中主动引入新注解体系
- 修改 Mapper、XML、SQL 参数或返回对象时，同步检查 Mapper 接口、XML、参数名、返回映射和调用方，不要只改一侧导致参数或映射失配
- 新增异常处理、关键分支或排查日志时，沿用项目已有日志方案；项目使用 Lombok 时可优先使用 `@Slf4j`；打印异常时保留完整堆栈和必要上下文变量，不要使用标准控制台输出替代日志

## 2.6 类注释和作者

新建 `.java` 类文件时，按用户或仓库约定补充类级文档注释，不要修改已有类似注释的内容。

需要生成作者注释时，优先读取当前目标仓库生效的 Git 用户名：`git config user.name`；无法获取时先询问用户或省略 `@author`。日期使用当前系统时间，精确到分钟。不要写死默认作者名、账号、邮箱或机器路径。

# 3. 测试与验证

## 3.1 测试策略

- 窄范围实现或修复：优先运行与修改范围最相关的单元测试、模块测试或编译命令，只扩大到必要模块
- 行为变化：补充聚焦单测，沿用当前模块已有测试框架和断言风格，不引入项目外的新测试风格
- 外部依赖参与逻辑：Mock 范围聚焦外部依赖。断言优先使用 AssertJ；如果项目已有统一断言工具，跟随项目

## 3.2 验证反馈

测试或编译已运行时，说明命令和结果。无法运行测试或编译时，说明原因，并列出已经完成的静态检查或人工核对范围。

回复用户时使用中文，简明说明修改了什么、验证了什么、是否存在未完成风险。
