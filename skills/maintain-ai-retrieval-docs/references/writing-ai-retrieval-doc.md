# 1. AI 检索文档写作要求

本文件只在准备生成或更新 AI 检索文档时读取。

# 2. 推荐结构

根据需求实际裁剪，优先使用：

- `# 1. 文档定位`
- `# 2. 核心结论`
- `# 3. 主要代码位置`
- `# 4. 执行入口链路`
- 领域行为章节，例如节点行为、转换规则、上下文字段、错误处理、持久化行为、元数据规则。
- 兼容性和边界章节。
- `# N. 当前测试重点` 或 `# N. 验证建议`
- `# N. AI 快速检索提示`

需要更完整模板时，读取 `document-template.md`。

# 3. 写作原则

- 面向后续排查和理解，不写成发布说明、汇报材料或产品介绍。
- 多写类名、方法名、常量名、文件路径、行为边界和验证命令。
- 记录隐藏规则和兼容性要求，例如默认值、开关、兜底逻辑、跳过分支、复用边界。
- 避免复制大段需求文档或代码。
- 不要编造未从需求、文档、代码、测试或 diff 中确认的行为。
- 不确定但重要的信息要明确标注，不要把推测写成事实。
- 优先写稳定事实，不写临时调试过程、未合并方案或个人判断。
- 旧逻辑、新逻辑和兼容行为同时存在时，明确说明默认路径和触发条件。

# 4. 风格参考

优先参考所在项目内已有的高质量 AI 检索文档，但不要把某个项目的目录名、模块名或历史需求写进通用规则。

评价参考文档是否可借鉴时，优先看这些特征：

- 是否清楚描述稳定代码事实、执行链路、兼容边界、验证方式和检索关键词。
- 是否区分已实现事实、需求目标和不确定信息。
- 是否避免复制需求正文、代码大段内容和临时讨论过程。
